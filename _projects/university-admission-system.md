### Overview

This project is a console-based University Admission Management System 
built in C# as a lab project for CSC-103L Object Oriented Programming 
(Spring 2026), supervised by **Miss Rimsha Chauhdary**.

The system manages the complete university admission lifecycle — from 
student application and merit list generation to subject registration 
and fee calculation — organized using a professional **Three-Tier 
Architecture** with separate Presentation, Business Logic, and Data 
layers.

The application features a polished animated console interface with 
typewriter text effects, a colored loading bar on startup, and 
role-specific color-coded portals — green for Student, yellow for 
Faculty, and magenta for Admin.

### Problem

University admission processes involve multiple stakeholders — students 
applying for programs, administrators managing merit lists and seat 
allocation, and faculty overseeing academic programs and subjects. 
Managing all of these in one system while enforcing strict business 
rules (merit-based allocation, credit hour limits, fee generation) 
requires careful software design.

The challenge was to implement this real-world system using proper 
Object Oriented Programming principles and a layered architecture — 
not as a single file, but as a structured, maintainable, multi-layer 
application designed from a UML class diagram first.

### Architecture

The system is built on a strict **Three-Tier Architecture:**

**Presentation Layer — Console Interaction**
Handles all user input and output through the console interface.
Classes: `Menu`, `StudentView`, `ProgramView`, `AdmissionView`
- Displays role-based menus with color-coded portals
- Takes user input for student data and program preferences
- Shows merit lists, program allocations, subjects, and fee details
- Calls functions from the Business Logic Layer only

**Business Logic Layer — Core Processing**
Handles all system rules, calculations, and data processing.
Classes: `Student`, `Program`, `Subject`, `AdmissionManager`,
`MeritCalculator`, `RegistrationManager`, `FeeCalculator`
- Calculates merit based on FSC and ECAT marks
- Generates and sorts merit lists by highest merit
- Allocates students to programs based on merit and preferences
- Enforces credit hour constraints (max 9 credit hours per student)
- Handles subject registration and fee calculation

**Data Layer — Storage**
Handles all in-memory data storage using Lists.
Classes: `StudentDL`, `ProgramDL`, `SubjectDL`
- Stores lists of students, programs, and subjects
- Provides Add, Retrieve, and Search methods
- Single source of truth for all application data

### Business Rules Enforced

- A program cannot have more than **20 credit hours** of subjects
- A student cannot register more than **9 credit hours**
- Merit list generated by **highest merit score + available seats**
- Fee calculated according to **registered subjects only**
- Students rank programs by **preference order**
- Admission assigned based on **merit + preference + seat availability**

### Merit Calculation

Merit is calculated from two academic scores:
- **FSC Marks** — Federal Science Certificate result
- **ECAT Marks** — Engineering College Admission Test score

Students are ranked by combined merit score. The Admin processes 
admissions starting from the highest merit student, respecting each 
student's program preferences and available seats. Students not 
allocated to any program are placed on a **waitlist**.

### OOP Concepts Demonstrated

- **Classes and Objects** — Student, Program, Subject, Department,
  AdmissionDept, FeeDept
- **Encapsulation** — private fields with public methods in all classes
- **Abstraction** — three-layer separation hides complexity from user
- **Association** — Program has Subjects, Student has Preferences
- **Lists and Collections** — in-memory data management throughout
- **UML Design First** — complete UML class diagram designed before 
  writing any code
- **Separation of Concerns** — each layer has exactly one 
  responsibility

### What I Learned

- How to design a complete system from a **UML class diagram** before 
  writing any code — understanding class relationships visually first
  made implementation significantly clearer
- How **Three-Tier Architecture** enforces separation — the UI never 
  touches data directly, and the data layer never knows about the UI
- How to implement **merit-based sorting and seat allocation** — a 
  real algorithm used in actual Pakistani university admission systems
- How to enforce **business rules at the correct layer** — credit hour 
  limits and merit calculations belong in the Business Logic Layer,
  not scattered across the presentation
- How to build a **role-based console application** where Student, 
  Faculty, and Admin users each see different menus and have 
  different permissions
- How to use **color-coded console output** in C# to create a 
  professional and readable terminal interface
- How to use **C# Lists and in-memory storage** to simulate a 
  database without an actual database engine
- The value of **seed data** — pre-loading sample programs and 
  students allowed immediate testing of all features without 
  manual data entry every run
